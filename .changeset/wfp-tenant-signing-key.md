Auto-generate a private signing key when provisioning a new WFP tenant, and fail
loudly when a tenant has none.

A freshly provisioned Workers-for-Platforms tenant only inherited the control
plane's **public** verify keys (private material is stripped at the boundary),
so it could serve discovery, `/authorize`, and JWKS but 500'd at `/oauth/token`
with nothing to sign — every first OIDC login against a new tenant failed until
a key was generated by hand.

- New exported `ensureSigningKey(keys, opts)` helper mints a create-if-missing
  RS256 `jwt_signing` key in a scope. It only mints when no *signable* key
  (private material present) exists, so it is idempotent across re-syncs. The
  private key is generated locally in the tenant DB and never crosses the
  control-plane boundary.
- The WFP tenant worker's `/internal/sync-defaults` handler now calls it after
  applying the projected defaults, so a brand-new tenant completes an
  Authorization-Code + PKCE login out of the box. The apply response surfaces
  `signingKey: { created }`. The projected public control-plane keys are kept
  (they verify forwarded control-plane admin tokens) — the tenant's own private
  key is added alongside them.
- `resolveSigningKeys`' sign path now selects the newest *signable* key (one
  with private material), skipping the verify-only public keys a tenant inherits.
  Without this, a control-plane key rotation could re-sync a newer public key
  that out-sorts the tenant's private key and reintroduces the 500.
- `/oauth/token` now returns a specific error naming the scope
  ("No signing key configured for tenant …") instead of an opaque 500 when a
  tenant somehow ends up with only public verify keys.

Fixes #1181.
